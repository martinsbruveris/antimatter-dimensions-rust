# Phase 4: Eternity (Features 4.1–4.6)

Design for the second prestige layer: Eternity Points, Eternity Milestones, Time
Dimensions, Time Studies, Eternity Challenges, and Eternity Upgrades. JS
reference: `src/core/eternity.js`, `src/core/time-theorems.js`,
`src/core/time-studies/`, `src/core/eternity-challenge.js`,
`src/core/dimensions/time-dimension.js`,
`src/core/secret-formula/eternity/*`,
`src/core/secret-formula/challenges/eternity-challenges.js`, and the
`FreeTickspeed` block of `src/core/tickspeed.js`.

Implementation order (dependency-driven, differs slightly from the feature
numbering): **4.1 Prestige → 4.2 Milestones → 4.3 Time Dimensions → 4.4 Time
Studies → 4.5 Eternity Challenges → 4.6 Eternity Upgrades**. Time Studies need
Time Dimensions (TT purchases are gated on owning a TD; studies unlock TD5–8);
ECs are entered through study slots; the `epMult` upgrade's effect slots into
the EP formula whenever it lands.

## 1. Feature 4.1 — Eternity Prestige

### Formulas

- Goal: `player.records.thisEternity.maxIP >= eternityGoal`, where
  `eternityGoal = 1e308` (`Decimal.NUMBER_MAX_VALUE`) outside an Eternity
  Challenge, else the EC's scaled goal (§5).
- EP gain (`gainedEternityPoints`, pre-Reality/Pelle terms dropped):

  ```
  EP = floor( 5 ^ (log10(thisEternity.maxIP + gainedInfinityPoints()) / 308 - 0.7)
              × totalEPMult )
  ```

  Note the *pending* crunch IP counts toward the exponent. `totalEPMult` is 1
  until its in-frontier sources land: `EternityUpgrade.epMult` (×5^purchases,
  §6), TS61 (×15), TS121/122/123 (§4).
- `gainedEternities()` = 1 (every other source is post-Reality).

### State (new fields on `GameState`)

| Field | JS | Notes |
|---|---|---|
| `eternity_points: Decimal` | `player.eternityPoints` | cumulative, spent on TDs/EU/TT |
| `eternities: Decimal` | `player.eternities` | drives milestones |
| `records.this_eternity: ThisEternity` | `records.thisEternity` | `time_ms`, `real_time_ms`, `max_am` (absorbs the existing `max_am_this_eternity` field), `max_ip` |
| `records.best_eternity: BestEternity` | `records.bestEternity` | `time_ms`, `real_time_ms` |

`thisEternity.maxIP` is advanced in `tick` (`max(maxIP, infinity_points)`) and
also **reset to 0 whenever IP is reset** (the original `Currency.infinityPoints
.reset()` also zeroes `thisEternity.maxIP` — i.e. on Eternity itself).
`eternity_unlocked` is derived on load like `infinity_unlocked`
(`eternities > 0 || EP > 0 || TT bought > 0 …`); we store a bool set on the
first eternity, mirroring how `infinity_unlocked` is handled.

### The reset (`eternity()`)

Reward phase (only when at goal — a *forced* eternity from EC entry/exit skips
it): lower `best_eternity` times, `EP += gained_eternity_points()`,
`eternities += 1`, EC completion when an EC runs (§5).

Reset phase, in original order (in-frontier slice):

1. `initializeChallengeCompletions`: clear NC + IC completions; if the
   `keepAutobuyers` milestone (2 eternities) is reached, re-complete all NCs
   (autobuyers come back); `autoIC` milestone (7) re-completes ICs as they
   unlock (in `tick`, §2). Clear `challenge.normal.current` /
   `challenge.infinity.current`.
2. `initializeResourcesAfterEternity`: `sacrificed = 0`, `infinities = 0`,
   `best_infinity = MAX` sentinel, `this_infinity` times = 0,
   `dim_boosts = keepInfinityUpgrades ? 4 : 0`,
   `galaxies = keepInfinityUpgrades ? 1 : 0`, `part_infinity_point = 0`,
   `infinity_power = 0`, `time_shards = 0`, `this_eternity` times = 0,
   `total_tick_gained = 0`, `eterc8ids = 50`, `eterc8repl = 40`.
3. `broke_infinity = false` unless the `keepAutobuyers` milestone is reached.
4. Infinity Dimensions **full** reset (amount/bought/cost/unlock all cleared).
5. `Replicanti.reset()`: back to defaults; stays unlocked (amount 1) iff the
   `unlockReplicanti` milestone (10) is reached.
6. `resetChallengeStuff`, AD reset, tickspeed reset.
7. Time-study respec if `respec` was ticked (§4); IP → 0 (also zeroing
   `thisEternity.maxIP`); ID amounts reset; time dimension amounts →
   bought base + cost rebuild (TDs themselves persist, §3).
8. `playerInfinityUpgradesOnReset`: keep all 16 infinity upgrades iff milestone
   4; keep the 12 break-infinity upgrades + rebuyables `[8,7,10]` iff milestone
   8; else clear (rebuyables `[0,0,0]`).
9. `this_infinity.max_am = 0`, `this_eternity.max_am = 0`, antimatter → starting
   value.

Autobuyer *configuration* (unlocks, modes) persists in the save either way; the
NC-completion clearing is what re-locks them pre-milestone-2, matching the
original (autobuyer unlock ≡ challenge completed).

### Save/load

New DTO fields (all already in every real save): `eternityPoints`, `eternities`
(Decimal strings), `records.thisEternity.{time, realTime, maxIP}` (`maxAM`
already read), `records.bestEternity.{time, realTime}`. Write path overlays the
same keys.

### UI

- Header: the original `EternityButton.vue` shows once `canEternity` (replacing
  the crunch button area); pre-first-eternity it reads "Other times await.. I
  need to become Eternal"; after, it shows the EP gain. Vendored classes
  `o-prestige-button--eternity` / `btn-eternity` etc.
- `EternityConfirmModal` mirroring the original `EternityModal.vue`
  (confirmation toggle `options.confirmations.eternity`).
- New **Eternity** top-level tab (conditional on eternity unlocked) with
  subtabs Time Studies (§4) / Milestones (§2) / Upgrades (§6); Time Dimensions
  live on the Infinity-Dimensions pattern (their own subtab, §3).
- Header currency line for EP (`Sidebar`/`GameHeader`), mirroring
  `CurrencyEternityPoints`.

## 2. Feature 4.2 — Eternity Milestones

Milestones are **pure derived state**: `EternityMilestone.isReached ⇔
eternities >= threshold`. No save data. The in-frontier list and where each
effect is applied:

| Eternities | id | Effect site |
|---|---|---|
| 2 | keepAutobuyers | eternity reset (§1 step 1/3) |
| 4 | keepInfinityUpgrades | reset step 2/8 |
| 7 | autoIC | per-tick: auto-complete an IC the moment it is unlocked |
| 8 | keepBreakUpgrades | reset step 8 |
| 10 | unlockReplicanti | reset step 5 |
| 25 | autoUnlockID | per-tick: `unlock_infinity_dimension` when reachable |
| 30 | unlockAllND | `unlocked_dimensions()` returns 8 |
| 40 | replicantiNoReset | `buy_replicanti_galaxy` skips the galaxy-style reset |

Deferred (documented, milestone still *displays* as reached): the autobuyer
milestones (1 `autobuyerIPMult`, 3 `autobuyerReplicantiGalaxy`, 5
`bigCrunchModes`, 9 `autobuyMaxGalaxies`, 11–18 ID autobuyers, 50/60/80
replicanti-upgrade autobuyers, 100 `autobuyerEternity`) target autobuyer types
we haven't built, and 6/200/1000 are offline-progress generators. They become
meaningful as the automation layer grows; the milestone grid shows them
faithfully either way.

UI: `EternityMilestonesTab` — the original's 3-column grid of
`EternityMilestone` cells (reached = green), driven by a snapshot list
`{ eternities, reward, is_reached }` with strings frontend-side, as for
achievements.

## 3. Feature 4.3 — Time Dimensions

### State

`time_dimensions: [TimeDimension; 8]` with `{ amount: Decimal, bought: u64,
cost: Decimal }` (save: `player.dimensions.time[]`), plus `time_shards:
Decimal` and `total_tick_gained: u64` (`player.totalTickGained`).

### Costs (`TimeDimensionState.nextCost`)

- Base costs `[1, 5, 100, 1000, 1e2350, 1e2650, 1e3000, 1e3350]` EP; per-buy
  multipliers `[3, 9, 27, 81, 24300, 72900, 218700, 656100]`.
- Tiers 5–8 below their `e6000ScalingAmount` (`[833, 689, 562, 456]` buys):
  plain `base × mult^bought`.
- Otherwise walk thresholds `[1.79e308, 1e1300, 1e6000]` with multiplier
  bumps `×[1, 1.5, 2.2]` — first threshold the cost fits under wins.
- Past `1e6000`: `base × (mult × 2.2 for tiers ≤ 4)^(scalingAmount + (bought -
  scalingAmount) × 4)` where tiers 1–4 use `e6000ScalingAmount`
  `[7322, 4627, 3382, 2665]`.

Purchases are single (`amount += 1`, `bought += 1`), cost recomputed from
`bought`; buy-max loops cheapest-first (`maxAllTimeDimensions` is approximated
by our usual repeated-single loop).

> **Ordering correction.** The feature-decomposition doc says TD5–8 are
> unlocked by "Time Studies 71–74". In the actual game they are unlocked by
> **Dilation studies 2–5** (`TimeStudy.timeDimension(tier) =
> DilationTimeStudyState.studies[tier - 3]`, costs 1e6–1e9 TT, gated on Time
> Dilation being unlocked) — i.e. **Phase 5 content**. Phase 4 therefore ships
> TD1–4 purchasable and TD5–8 present in state/save but locked; the dilation
> gate arrives with Feature 5.1.

### Multiplier & production

`multiplier(tier) = commonMult × 4^bought` (tier-8 `bought` capped at 1e8),
where `commonMult` collects (in-frontier): TS11 (tier 1 only), TS73 (tier 3),
TS93, TS103, TS151, TS221, EC1/EC10 rewards, EU4–6 (`tdMultAchs`,
`tdMultTheorems`, `tdMultRealTime`). TD8→TD7→…→TD1 at `diff/10`; TD1 produces
Time Shards at `diff`. EC7/EC11 modify the chain (§5).

### Free tickspeed (`FreeTickspeed.fromShards`)

```
tickmult = 1.33 (TS171 → 1.25)
uncapped = max(0, ln(shards) / ln(tickmult))
if uncapped <= 300000: gained = ceil(uncapped)
else: soft-capped region — cost(bought) = c·bought² + bought (c =
      GROWTH_RATE/GROWTH_EXP/ln(tickmult)), solved by Newton's method
```

`total_tick_gained` = the resulting count; **tickspeed totalUpgrades = bought +
gained**, so `tickspeed_effect` must move to `Decimal` (with thousands of free
upgrades `0.8^n` underflows `f64`). `current_tickspeed_ms` becomes a
`Decimal`-returning function; callers adjust.

### UI

`TimeDimensionsTab` mirroring the original `TimeDimensionsTab.vue` (Modern):
EP header, per-row amount/mult/cost, "Buy One"/"Buy Max", time-shard +
free-tickspeed readout.

## 4. Feature 4.4 — Time Studies

### Time Theorems

`timestudy` save slice: `theorem: Decimal`, `maxTheorem: Decimal`, `amBought`,
`ipBought`, `epBought` (purchase counts), `studies: []`. Costs: AM `1e20000 ×
1e20000^n`, IP `1 × 1e100^n`, EP `1 × 2^n`. Buying requires owning ≥1 TD
(`TimeTheorems.checkForBuying`). Buy-one and buy-max per currency.

### Studies

`studies: Vec<u16>` (bought normal studies, original order preserved), plus
`respec: bool` and the EC slot (`challenge.eternity.unlocked`, §5). A study is
buyable iff: not bought, `theorem >= cost`, and its `requirement`/`reqType`
holds:

- `AT_LEAST_ONE` / `ALL` over listed study ids (+ closures like "EC completion
  > 0").
- `DIMENSION_PATH` (studies 71/72/73): all listed prereqs **and** at most
  `1 + TS201` distinct dimension paths among the bought 71/72/73.
- Pace split: 121/122/123 (and their followers 131–133 / 141–143) are mutually
  exclusive columns pre-Ra: buying into one column blocks the other two (the
  original enforces via `requiresST` — without Space Theorems only one of the
  set can be bought).
- 221–234 pairs (e.g. 221/222) are likewise ST-locked to one of each pair.
- Triad studies 301–304 are Ra content: **out of frontier**.

In-frontier effects wired at their engine sites (see the JS table in
`normal-time-studies.js`): TS11 (TD1 mult from tickspeed), TS21/22 (replicanti
formula/interval floor), TS31 (infinity-count effects ^4), TS32 (×max(boosts,1)
infinities), TS33 (keep half RGs on crunch), TS41 (1.2^all-galaxies IP mult),
TS42 (galaxy requirement step 60→52), TS51 (×1e15 IP), TS61 (×15 EP), TS62 (×3
replicanti speed), TS71/72/73 (sacrifice → AD/ID4/TD3, capped), TS81–83
(dimboost power ×10 / boosts²→ID / tick-gained→boost mult), TS91–93 (this-
eternity-time AD mult / best-eternity ID mult / tick-gained TD mult), TS101–103
(replicanti AD mult / RG replicanti mult / RG TD mult), TS111 (IP formula
divisor 308→285), TS121–123 (EP ×(avg eternity speed | 35 | idle-time)),
TS131–133 (RG cap +50% / RGs 40% stronger + replicanti ×1.5? (perk-gated
variants dropped) / RGs 50% stronger but replicanti ×10 slower), TS141–143 (IP
mult decaying ×1e45 / flat 1e25 / growing), TS151 (×1e4 TD), TS161/162
(×1e616 AD / ×1e11 ID), TS171 (free-tickspeed mult), TS181 (passive IP: 1% of
crunch gain per second; requires EC1–3 completions), TS191 (banked infinities:
+5% on eternity — new `infinities_banked: Decimal` state; `infinitiesTotal =
infinities + banked` feeds TS31-boosted effects and EC4/EC10), TS192
(replicanti uncap → post-cap slow growth), TS193 (AD mult 1e13000^min(etern/1e6,
1)), TS201 (second dimension path), TS211–214, TS221–234 (each pair one-side-
only pre-Ra).

TS121's "average of last 10 eternities" needs `records.recentEternities`; we
model the original's 10-entry ring of `[time, realTime, EP, eternities-gained]`.

Respec: refund all normal studies + the EC slot; fires on the next eternity
(flag in save: `player.respec`).

### UI

`TimeStudiesTab`: the vendored tree (study buttons `o-time-study`,
type-coloured: normal/dark/light per row class), SVG connection lines
(`time-study-connections.js` list), TT header with the three buy buttons +
respec toggle. Study descriptions frontend-side; engine ships per-study
`{ id, cost, bought, available }` + TT/generation info.

## 5. Feature 4.5 — Eternity Challenges

### State

- `challenge.eternity.current: u8` (0 = none), `challenge.eternity.unlocked:
  u8` (the *one* EC whose study is currently held; 0 = none).
- `eternity_challenges: [u8; 12]` completions (save: `eternityChalls` map
  `"eterc<N>" → count`), max 5 each.
- `eterc8ids: i32`, `eterc8repl: i32` (EC8 purchase budgets, reset to 50/40 on
  eternity).

### Flow

Unlock: buy the EC study (TS ids 181/EC1 … per `ec-time-studies.js`; cost in
TT + a **secondary requirement**, e.g. EC1 "20000 + 20000/completion
eternities", EC5 "160+14/c galaxies"…). Only one EC study can be held at a
time; respec refunds it. Start: forced eternity into the challenge
(`startEternityChallenge` reset — same as §1 but without rewards and keeping
dilation flags out pre-Phase-5). Goal: `thisEternity.maxIP >=
goal × goalIncrease^completions`. Completing: on the eternity that banks it,
`completions += 1`, EC study slot consumed (`unlocked = 0`), auto-respec.
Exit/failure (EC4: `infinities > max(16-4c, 0)`; EC12: `thisEternity.time >
(10-2c)/10 s`) → forced eternity out.

### Restrictions & rewards (in-frontier wiring sites)

| EC | Restriction site | Reward site |
|---|---|---|
| 1 | TD production = 0 | TD mult `max(thisEternityMs/10, 0.9)^(0.3+0.05c)` |
| 2 | ID production = 0 | ID1 mult `infinityPower^(1.5/(700-100c))`, cap 1e100 |
| 3 | AD5–8 produce nothing; sacrifice disabled | buy-10 mult `+0.72c` |
| 4 | infinity mults/generators off; ≤ max(16-4c,0) infinities | ID mult `IP^(0.003+0.002c)`, cap 1e200 |
| 5 | galaxy distant scaling from 0; dimboost cost `+e3` scaling | distant galaxy start `+5c` |
| 6 | no antimatter galaxies purchasable; RG-cap upgrade cheap | AD cost mult growth `-0.2c` |
| 7 | TD1 → ID8, ID1 → AD7; tickspeed hits ID+TD | TD1 produces ID8 (`prod^0.2c - 1`/s) |
| 8 | 50 ID + 40 replicanti purchase budget | RGs stronger by infinity power |
| 9 | tickspeed purchases disabled; infinity power → TD (reduced) | ID mult `timeShards^0.1c`, cap 1e400 |
| 10 | TD+ID disabled; AD ×`infinitiesTotal^950` | TD mult `(2.783e-6·total)^(0.4+0.1c)` |
| 11 | only IP-power + dimboost AD mults | tickspeed cost mult growth `-0.07c` |
| 12 | game speed ×0.001; time limit `(10-2c)/10 s` | ID cost mults `^(1-0.008c)` |

EC11/EC12 introduce a **game-speed factor** hook (`getGameSpeedupFactor`):
`tick` splits real `dt` from game `dt` (EC12 multiplies game-dt by 1e-3). This
is the first game-speed consumer; records already track game vs real ms.

UI: `EternityChallengesTab` under Challenges (12 `c-challenge-box` tiles with
completion counts ×5, goal display, unlock-study gating).

## 6. Feature 4.6 — Eternity Upgrades

State: `eternity_upgrades: u32` bitmask (save: `eternityUpgrades` Set of ids
1–6) + `epmult_upgrades: u32` (save: `epmultUpgrades`).

| id | Cost | Effect |
|---|---|---|
| 1 `idMultEP` | 5 EP | ID mult `unspent EP + 1` |
| 2 `idMultEternities` | 10 EP | ID mult `(eternities/200+1)^log4(2·eternities+1)` (softcap 1e5) |
| 3 `idMultICRecords` | 5e4 EP | ID mult `2^(30/max(sum IC times s, 0.1))`, cap `2^30×0.61` |
| 4 `tdMultAchs` | 1e16 EP | TDs × achievement power |
| 5 `tdMultTheorems` | 1e40 EP | TDs × max(unspent TT, 1) |
| 6 `tdMultRealTime` | 1e50 EP | TDs × max(days played, 1) |

Row-1 upgrades (1–3) require nothing; each row-2 upgrade requires the one above
it (original grid: 2 rows × 3, `SetPurchasableMechanicState` with no prereqs —
verify at implementation; the original UI shows a plain 2×3 grid, all
independently purchasable).

`epMult`: rebuyable ×5 EP-mult. Cost `500 × stepMult^count` where `stepMult`
walks `[50, 100, 500, 1000]` against thresholds `[1e100, 1.79e308, 1e1300,
1e4000]`, then `500 × 1e3^(count + max(count-1334,0)^1.2)`. Effect feeds
`totalEPMult` (§1). Buy-one + buy-max (repeated singles).

Upgrade 3 needs **IC completion times**: add `infinity_challenge_best_times:
[f64; 8]` (save: `challenge.infinity.bestTimes`), recorded on IC completion in
`big_crunch_reset`.

UI: `EternityUpgradesTab` — original `EternityUpgrades.vue` layout (2×3 grid of
`o-eternity-upgrade` buttons + the epMult box with buy-max).

## 7. Cross-cutting notes

- **Game-speed factor.** EC12 forces a global ×0.001. `tick(dt_ms)` gains an
  internal `game_dt = dt_ms × game_speed()`; records split real/game time
  accordingly. Pre-EC12 `game_speed() == 1`, so nothing changes.
- **Decimal tickspeed.** Free tickspeed upgrades push `0.8^n` past f64;
  `current_tickspeed_ms`/`tickspeed_effect` move to `Decimal` internally.
- **infinitiesTotal.** `infinities + infinities_banked` — used by TS31-affected
  effects, EC10, and the `infinitiedMult` break upgrade.
- **Save fidelity.** All new fields exist in every real save (fresh-start
  template already carries them); DTO additions are required fields, overlay
  writes them back. `eternityChalls` is an object map, `eternityUpgrades` /
  `studies` arrays, all plain.
- **Frontier.** Dilation studies (TS225+ dilation row), triads (301–304), Space
  Theorems, perks, and every Reality+ term in the JS formulas are dropped, each
  noted at its site.
